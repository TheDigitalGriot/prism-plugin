# Spectrum Autonomous Execution Workflow

## Research Question

How does the Spectrum autonomous execution workflow function in this project? What are the components of spectrum.sh and the prism-spectrum skill, and how do they interact?

## Summary

Spectrum is a multi-layer autonomous execution system that decomposes approved Prism plans into atomic stories and executes them one-per-session in a loop. The shell-level orchestrator (`scripts/spectrum.sh`) spawns fresh Claude CLI sessions iteratively, while the `prism-spectrum` skill defines the in-session workflow for executing a single story. The system uses a signal protocol (`<spectrum-continue>`, `<spectrum-retry>`, `<spectrum-blocked>`, `<spectrum-error>`, `<promise>COMPLETE</promise>`) for communication between the per-session AI and the orchestration loop. Three independent implementations exist: a bash script, a Go TUI dashboard, and a TypeScript VSCode/Electron engine.

## Component Analysis

### 1. Pipeline Overview

The full Spectrum pipeline follows this sequence:

```
/prism-plan  -->  /decompose_plan  -->  spectrum.sh (or CLI/VSCode)
   (plan)         (stories.json)        (iterative execution loop)
```

Each iteration within the loop invokes the `prism-spectrum` skill inside a fresh Claude session.

### 2. spectrum.sh (Bash Orchestrator)

**Location**: `scripts/spectrum.sh:1-313`

The script is the original orchestrator implementation. It operates as follows:

**Prerequisites** (`spectrum.sh:75-94`):
- Requires `claude` CLI and `jq` to be installed
- Requires a `stories.json` file (generated by `/decompose_plan`)

**Configuration** (`spectrum.sh:34-66`):
- `STORIES_FILE`: defaults to `$PROJECT_DIR/.prism/stories/stories.json`, or accepts a path argument
- `PROGRESS_FILE`: derived from the stories path via `derive_progress_path()` at line 41
- `MAX_ITERATIONS`: env var `SPECTRUM_MAX_ITERATIONS`, default 50
- `VERBOSE`: env var `SPECTRUM_VERBOSE`, default false
- `PAUSE`: env var `SPECTRUM_PAUSE`, default 2 seconds

**Progress path derivation** (`spectrum.sh:41-61`):
- Flat layout: `.prism/stories/stories.json` maps to `.prism/shared/spectrum/progress.md`
- Epic-scoped layout: `.prism/stories/<epic>/stories.json` maps to `.prism/shared/spectrum/<epic>/progress.md`

**Main loop** (`spectrum.sh:212-308`):
1. Checks prerequisites and initializes progress file
2. Enters a `while` loop bounded by `MAX_ITERATIONS`
3. Each iteration calls `run_iteration()` which invokes `claude --dangerously-skip-permissions --print` with the prompt: `"Execute the next story from $STORIES_FILE using the /prism-spectrum workflow. Progress file: $PROGRESS_FILE"`
4. Output is parsed for signals via `check_signals()` using `grep`
5. Signal determines next action: continue, retry, stop, or break

**Signal checking** (`spectrum.sh:176-209`):
- Uses `grep -q` to search for signal tags in output
- Returns exit codes: 0=complete, 1=continue, 2=retry, 3=error
- `<spectrum-blocked>` is treated as continue (return 1) to try the next available story

**Error handling** (`spectrum.sh:276-279`):
- Tracks `consecutive_errors` counter
- Stops after `max_consecutive_errors` (hardcoded to 3)

**Iteration management** (`spectrum.sh:154-173`):
- In verbose mode, output is teed to stderr for real-time visibility
- Non-verbose mode captures output silently

### 3. prism-spectrum Skill (In-Session Workflow)

**Location**: `skills/prism-spectrum/SKILL.md:1-407`

This skill defines what happens inside each Claude session spawned by the orchestrator.

**Philosophy** (SKILL.md:12-18):
1. Fresh Start: load all context from files, never assume prior context
2. One Story: execute exactly one story per invocation
3. Quality Gates: must pass typecheck/lint/test before commit
4. Atomic Commits: one story = one commit
5. Learn Forward: capture learnings for future iterations

**Workflow steps**:

**Step 1 - Load State** (SKILL.md:36-49): Read stories.json, progress.md, and CLAUDE.md completely.

**Step 1b - Load Epic + Story Context** (SKILL.md:51-62): Extract `epic.decisions`, `epic.risks`, `epic.outOfScope`, `epic.references`, and per-story `context.why`, `context.risks`, `context.patterns`, `context.edgeCases`. This step exists in the current skill version but not in the v2.4.8 snapshot.

**Step 1c - Graph Verification** (SKILL.md:64-79): If codebase-memory-mcp is available, run `index_repository` and `trace_call_path` for `graphTargets` before implementing, then re-index afterward and check for dead code.

**Step 2 - Check Completion** (SKILL.md:81-89): If no incomplete stories remain, output `<promise>COMPLETE</promise>` and exit immediately.

**Step 3 - Pick Next Story** (SKILL.md:91-104): Filter stories by status != complete, filter out blocked stories (blockedBy references an incomplete story), sort by priority ascending, pick first.

**Step 4 - Announce Story** (SKILL.md:106-117): Output `<spectrum-story>` tag with ID, title, priority, and files.

**Step 5 - Implement Story** (SKILL.md:119-132): Read all files in the story's `files` array before making changes. Follow story steps, mark each as done. Follow existing code patterns from progress.md.

**Step 6 - Run Quality Gates** (SKILL.md:134-157): Execute commands from `epic.qualityGates` (e.g., `npm run typecheck`, `npm run lint`, `npm test`). If any gate fails: do NOT commit, capture error, run auto-debug investigation, record in progress.md, output `<spectrum-retry>`.

**Step 6b - Browser Verification** (SKILL.md:159-181): If UI files were modified and playwright-cli is available, start dev server, take screenshot, check console errors. Optional step.

**Step 7 - Commit Changes** (SKILL.md:183-195): `git add` specific files, `git commit` with `[STORY-XXX] [title]` format.

**Step 8 - Update State Files** (SKILL.md:197-229): Set story status to "complete", set `completedAt` and `commitHash`. Append entry to progress.md with summary, learnings, files changed, and quality gate results.

**Step 9 - Signal Continuation** (SKILL.md:231-258): Re-read stories.json and explicitly count remaining stories. If remaining > 0: output `<spectrum-continue>`. If remaining === 0: output `<promise>COMPLETE</promise>`. This step emphasizes re-reading the file to avoid premature completion signals.

**Debug Integration** (SKILL.md:262-386): When quality gates fail, three debug agents are spawned in parallel: `log-investigator`, `state-investigator`, `git-investigator`. Their findings are synthesized into a root cause hypothesis and suggested fix, recorded in progress.md, and included in the `<spectrum-retry>` signal for the next iteration.

### 4. Signal Protocol

**Definition**: `cmd/prism-cli/domain/signals.go:1-191`

Five signal types are defined as constants (`signals.go:11-18`):

| Signal | Tag | Go Constant | Purpose |
|--------|-----|-------------|---------|
| Complete | `<promise>COMPLETE</promise>` | `SignalComplete` | All stories done, terminate loop |
| Continue | `<spectrum-continue>...</spectrum-continue>` | `SignalContinue` | Story done, more remain |
| Retry | `<spectrum-retry reason="...">...</spectrum-retry>` | `SignalRetry` | Recoverable error, retry in fresh session |
| Blocked | `<spectrum-blocked reason="...">...</spectrum-blocked>` | `SignalBlocked` | Story blocked, skip to next |
| Error | `<spectrum-error reason="...">...</spectrum-error>` | `SignalError` | Fatal error, stop loop |

`ParseSignal()` at `signals.go:56-101` uses regex matching with a priority order: complete > error > retry > blocked > continue. The `reason` attribute is extracted separately via `extractReason()` at `signals.go:104-117`.

`ParseStoryAnnouncement()` at `signals.go:128-175` extracts story metadata from `<spectrum-story>` tags (ID, title, priority, files list).

### 5. stories.json Schema

**Source**: `commands/decompose_plan.md:129-244`

The stories.json file has two top-level keys:

**`epic`** object:
- `name`: plan title
- `source`: path to the source plan
- `createdAt`: ISO timestamp
- `qualityGates`: array of verification commands
- `decisions`: architectural decisions from the plan
- `references`: reference implementations
- `outOfScope`: items explicitly excluded
- `risks`: brief risk summaries

**`stories`** array, each story containing:
- `id`: `STORY-XXX` format
- `title`, `description`: human-readable text
- `priority`: integer (lower = done first; 1-10 foundation, 11-20 core, 21-30 integration, 31-40 tests, 41-50 polish)
- `status`: `"pending"`, `"in_progress"`, or `"complete"`
- `blockedBy`: null or `STORY-XXX` reference
- `files`: array of `{path, action}` objects (action: create/modify/delete)
- `steps`: array of `{description, done}` objects
- `context` (optional enrichment): `{why, risks, edgeCases, patterns, graphTargets}`

### 6. Progress File (progress.md)

**Go implementation**: `cmd/prism-cli/domain/progress.go:1-217`

The `ProgressFile` struct manages the progress.md file. `NewProgressFile()` at `progress.go:21-41` derives the path from the stories.json path using the same flat/epic-scoped logic as spectrum.sh.

The file contains:
- YAML frontmatter with `epic`, `startedAt`, `lastUpdated`
- A "Codebase Patterns (Consolidated)" section at the top for accumulated learnings
- Per-iteration entries appended via `AppendEntry()` at `progress.go:76-113`, each with timestamp, story ID, summary, learnings, files changed, and quality gate results

`ReadPatterns()` at `progress.go:173-216` parses the consolidated patterns section by finding content between the `## Codebase Patterns (Consolidated)` header and the next `---` separator.

### 7. Go TUI Dashboard (SpectrumPlugin)

**Location**: `cmd/prism-cli/app/plugin_spectrum.go:1-1219`

The `SpectrumPlugin` struct at `plugin_spectrum.go:60-118` implements a full Bubble Tea dashboard with:
- Story list with pagination (12 per page)
- Activity panel showing current tool and recent activities
- Log panel with paginated output
- Spring-based animations via Harmonica (progress bar overshoot, story completion pop, log entry slide-in, prism ray animation, shimmer)

**State machine** (`plugin_spectrum.go:773-867`): States include `StateIdle`, `StateRunning`, `StatePaused`, `StateComplete`, `StateMaxIterations`, `StateError`. The `handleSignal()` method processes signals identically to spectrum.sh but with additional safeguards -- notably at line 822, it detects and ignores premature `COMPLETE` signals when remaining stories > 0.

**Execution loop** (`plugin_spectrum.go:339-365`): The `StartNextIterationMsg` handler increments the iteration counter, creates an output channel, and calls `claude.RunClaudeStreamingCmd()` to spawn a new Claude process. Upon `ClaudeFinishedMsg`, it parses signals via `domain.ParseSignal()` and dispatches to `handleSignal()`.

**Epic support** (`plugin_spectrum.go:488-514`): The dashboard supports multiple epics via `DiscoverEpicsCmd` and `EpicSelectedMsg`, allowing tab-based switching between epic-scoped stories files.

**Demo mode** (`plugin_spectrum.go:392-486`): A simulation mode that cycles through stories with timed completions and random tool activity messages, used for showcasing the dashboard without executing real Claude sessions.

### 8. TypeScript Engine (VSCode/Electron)

**SpectrumEngine** (`packages/prism-core/src/core/controller/prism/spectrum.ts:1-353`):

A state machine class managing execution state with the same state set: idle, running, paused, complete, maxIterations, error. Key methods:
- `start()` / `pause()` / `resume()` / `stop()` at lines 159-201
- `incrementIteration()` at line 245 returns false when cap is exceeded
- `recordSignal()` at line 268 tracks signal type and manages consecutive error count
- `hasTooManyErrors()` at line 289 checks against the configured threshold (default 3)
- Elapsed time tracked via a 1-second `setInterval` timer

The state is broadcast to webviews via the `_onStateChange` callback, serialized as the `SpectrumState` interface (line 41-63) containing execution state, progress percentage, elapsed time, recent activities (last 50), and logs (last 200).

**SpectrumRunner** (`packages/prism-core/src/core/controller/prism/spectrum-runner.ts:1-259`):

Manages single-iteration execution via `PluginBridge`. The `runIteration()` method at line 95:
1. Gets next story from `StoriesManager`
2. Marks it `in_progress` with a UUID session ID
3. Calls `PluginBridge.executeSpectrum()` to spawn Claude CLI
4. Parses signal from output via `parseSignal()`
5. Updates story status: complete signals mark story complete; blocked/retry reload fresh state; errors keep story as in_progress

Events emitted include `story_started`, `story_complete`, `story_blocked`, `story_retry`, `story_error`, `all_complete`, `no_next_story`, `tool_activity`, and `log`.

### 9. decompose_plan Command

**Location**: `commands/decompose_plan.md:1-306`

Converts an approved plan into stories.json format. Key aspects:
- Analyzes phase complexity to determine single vs. multi-story splitting (line 28-40)
- Story sizing guide: "if you can't describe the implementation in 2-3 sentences, split it" (line 59)
- Priority ranges: 1-10 foundation, 11-20 core, 21-30 integration, 31-40 tests, 41-50 polish (line 76-84)
- Sets `blockedBy` when stories have file or type dependencies (line 63-72)
- Extracts quality gates from the plan's verification commands (line 86-103)
- Presents decomposition for user review before generating files (line 169-210)
- Optionally generates comparison files for A/B token measurement (`stories-v1.json`, `stories-v2.json`) at line 247-252

## Patterns Found

### Signal Communication Pattern
All three implementations (bash, Go, TypeScript) use the same XML-like signal tags for communication between the per-session AI and the orchestration loop. The bash script uses `grep -q` (`spectrum.sh:180-208`), the Go implementation uses compiled regex patterns (`signals.go:46-53`), and the TypeScript implementation uses `parseSignal()` imported from `@prism-core/prism/signals` (`spectrum-runner.ts:18`).

### Fresh Context Pattern
The core design principle is that each iteration runs in a completely fresh Claude session. Memory persists exclusively through files: stories.json (story status), progress.md (accumulated learnings), and git commits (code changes). This is enforced by spectrum.sh spawning a new `claude` process each iteration (`spectrum.sh:165-167`) and by the prism-spectrum skill's Rule 1: "Load state fresh - Never assume prior context, always read files" (`SKILL.md:252`).

### Path Derivation Pattern
The flat-to-epic path derivation logic appears identically in three places:
- `spectrum.sh:41-61` (bash `derive_progress_path()`)
- `cmd/prism-cli/domain/progress.go:21-41` (Go `NewProgressFile()`)
- The prism-spectrum skill documents both layouts at `SKILL.md:29-31`

### Consecutive Error Guard
All three implementations cap consecutive errors at 3 before stopping:
- `spectrum.sh:233,276-279` (`max_consecutive_errors=3`)
- `plugin_spectrum.go:75` (`maxConsecutiveErrs: 3`)
- `spectrum.ts:99` (`maxConsecutiveErrors: 3`)

### Premature Completion Guard
The Go TUI dashboard includes an additional safeguard not present in spectrum.sh: at `plugin_spectrum.go:821-827`, it checks `remainingCount()` after receiving a `SignalComplete` and ignores the signal if stories remain, logging a warning about a potential skill bug.
