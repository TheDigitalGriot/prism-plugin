# Story Manifest Schema

A `story-manifest.json` file provides per-requirement tracking for a story, enabling cross-session progress persistence and granular quality gate verification.

## Location

- Flat: `.prism/stories/<story-id>-manifest.json`
- Epic-scoped: `.prism/stories/<epic>/<story-id>-manifest.json`

## Schema

```json
{
  "story_id": "STORY-001",
  "requirements": [
    {
      "id": "REQ-001",
      "description": "Create the database migration file",
      "depends_on": [],
      "owns_files": ["src/db/migrations/001_create_users.ts"],
      "gate": "npm test -- --grep 'migration'",
      "contracts_to_read": [],
      "contracts_to_write": ["interfaces.json"],
      "passes": false
    },
    {
      "id": "REQ-002",
      "description": "Add indexes for query optimization",
      "depends_on": ["REQ-001"],
      "owns_files": ["src/db/migrations/002_add_indexes.ts"],
      "gate": "npm test -- --grep 'indexes'",
      "contracts_to_read": ["interfaces.json"],
      "contracts_to_write": [],
      "passes": false
    }
  ],
  "last_session": {
    "timestamp": "2026-03-07T14:30:00Z",
    "commit": "abc1234",
    "requirements_passing": 1,
    "requirements_total": 2
  }
}
```

## Field Reference

### Top-level

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `story_id` | string | yes | Must match the `id` field in `stories.json` |
| `requirements` | array | yes | Ordered list of requirements for this story |
| `last_session` | object | no | Populated after each session that works on this story |

### Requirement

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `id` | string | yes | Unique within this manifest (e.g., `REQ-001`) |
| `description` | string | yes | What this requirement accomplishes |
| `depends_on` | string[] | yes | IDs of requirements that must pass first (empty array if none) |
| `owns_files` | string[] | yes | Files this requirement creates or modifies |
| `gate` | string | yes | Command to verify this requirement passes |
| `contracts_to_read` | string[] | no | Contract files in `.prism/shared/contracts/` this requirement consumes |
| `contracts_to_write` | string[] | no | Contract files this requirement produces or updates |
| `passes` | boolean | yes | Whether the gate command last succeeded |

### Last Session

| Field | Type | Description |
|-------|------|-------------|
| `timestamp` | string (ISO-8601) | When the session ran |
| `commit` | string | Short git hash of the session's commit |
| `requirements_passing` | number | Count of requirements where `passes: true` |
| `requirements_total` | number | Total requirements in the manifest |

## Usage

### In `prism-spectrum`

When a manifest exists for the current story:

1. Read the manifest before implementing
2. Skip requirements where `passes: true` (already done in a previous session)
3. Implement one requirement at a time, respecting `depends_on` ordering
4. After each requirement, run its `gate` command
5. Update `passes: true` on success
6. Update `last_session` before signaling

### Generation

Manifests are generated by `/decompose_plan` (Step 7). Each story step maps to a requirement. The `gate` field defaults to the epic's `qualityGates` if no step-specific gate is available.

### Fallback

If no manifest exists for a story, `prism-spectrum` falls back to standard behavior (implement all steps sequentially, run quality gates at the end).
